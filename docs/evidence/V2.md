# V2 Evidence — Intent-based conventional interaction

## Intent semantics (documented, per exit criteria)

- The vocabulary is five **instant** intents: `select-photo`, `move-focus`,
  `open-focused`, `back`, `reset-view` (`src/input/intent.ts`).
- **No lifecycle wrappers exist** because no genuinely continuous intent exists yet;
  camera orbit remains a camera-rig concern (explicit V2 non-goal honoured).
- Intents are requests: `state/intentBindings.ts` is the single guard point and ignores
  intents that do not apply to current state (e.g. `move-focus` while a detail view is
  open). This is what makes repeated/competing input and cancellation safe per device.
- No source-arbitration framework exists — there is no demonstrated competing-source
  case (exit criterion honoured by absence).

## Input matrix — verified 2026-07-15

| Action | Pointer (mouse) | Keyboard | Touch/DOM |
|---|---|---|---|
| Browse / move focus | orbit + hover | ← → ↑ ↓ (live-verified) | scroll cards (fallback) |
| Open detail | click print (raycast → same emit; V1-verified pathway) | Enter via `open-focused` (live-verified) | card tap (live-verified) |
| Read metadata | ✓ | ✓ | ✓ |
| Back | "← Back" button (live-verified) | Escape (live-verified) | same button |
| Reset view | HUD button | R | n/a in fallback |

Guard behaviour verified live: `move-focus` emitted while a detail view was open did not
change the focused photo; `back` with nothing selected is a no-op.

## Architecture import review — verified 2026-07-15

- `src/input/**` imports nothing from `gallery/`, `scene/`, `ui/`, or `state/` (grep: no matches).
- `src/scene/**` imports nothing from `gallery/` or `ui/` (grep: no matches).
- `src/gallery/**` never imports `data/` directly (grep: no matches).
- Replacing an input adapter therefore requires zero gallery or scene changes: adapters
  only know the bus.
