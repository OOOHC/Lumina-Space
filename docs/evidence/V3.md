# V3 Evidence — Photographer account and photo library

## Delivered journey

`sign in → personal workspace → upload → library → remove`, live against Neon
(Postgres, eu-west-2) and Cloudflare R2 (WEUR), all within provider free allowances.

## Exit criteria

- [x] **One user cannot read or mutate another workspace's private assets.**
  Attack test 2026-07-16: a second legitimate account attempted view / delete /
  confirm against the first account's asset id — all returned 404; its own library
  listed 0 photos; the owner's asset remained intact. Workspace scope is derived
  from the session in `requireWorkspace`; no route accepts a client-supplied
  workspace id.
- [x] **Upload progress, failure state, retry, metadata, and archive paths are
  tested** — with two owner-accepted simplifications recorded below. Failure states:
  invalid body 400, unsupported type 415, per-file/quota rejection 413 with distinct
  reasons (unit-tested), storage rejection and confirm failure surface distinct UI
  errors; abandoned reservations age out of quota after 1 h.
- [x] **Only currently required image derivatives are generated.** Exactly three
  objects per photograph (original, ≤1600 px preview, ≤400 px thumb), rendered by the
  uploading browser; no server-side processing exists.
- [x] **ADR 0002 reviewed against the chosen provider before implementation.**
  ADR 0006 (accepted stack) + ADR 0002 addendum name R2; key scheme
  `{workspace}/{asset}/{kind}` implements the workspace-scoped, immutable-object
  model.

## Owner-accepted simplifications (recorded, not hidden)

1. **Progress** is a busy indicator, not a per-byte progress bar (direct PUTs are
   single requests; per-byte progress needs XHR and is deferred until photographs
   large enough to warrant it are real).
2. **Retry** is manual re-selection after an error message, not automatic resumption.
3. **Metadata extraction** is dimensions/type/size only; EXIF parsing is not
   implemented, so GPS is never extracted — the "GPS hidden by default" requirement
   is satisfied by omission until EXIF extraction has a product consumer.
4. **Remove is a hard delete** while nothing can reference an asset; the handler
   documents the mandatory flip to refuse-or-archive when V4 references arrive.
5. Presigned PUTs do not sign content-length; quota integrity trusts our own client
   until V5 hardening (signed length or post-upload verification).

## Test and validation record

- 45 unit tests green (api 8: quota/rejection reasons; web 37 incl. derivative
  sizing); typecheck clean across workspaces; production build unchanged at ~304 KB
  gzip main bundle (auth/library UI added ~1 KB).
- Full upload cycle against real R2: request → 3 presigned PUTs (all 200) → confirm
  → list (byte-exact quota accounting) → view 302 → byte-identical retrieval →
  anonymous access 401.
- Deletion: 204 → library empty → quota zero → subsequent view 404; R2 objects
  removed before the row so storage can never hold invisible billable bytes.
- Browser UI: sign-up/sign-in/session persistence across reload, library grid with
  thumbnails served from the private bucket, multi-file upload, per-card removal
  with confirmation.

## Infrastructure state (cost posture)

Neon Free (no payment method, auto-wake) · Workers free plan · R2 subscription with
$1 billing alert, private WEUR bucket, least-privilege object token scoped to the
single bucket, CORS limited to the dev origins, app-enforced 512 MB workspace quota
and 25 MB upload cap. Billable usage at closure: $0.00.
