# V2.5 Evidence — Gesture Experience Validation

## What is implemented

The Curator's Hand sequence (DESIGN.md) as a pure input adapter:

- **Explicit opt-in only:** gesture mode starts from a HUD button; there is no camera
  request on page load (non-goal honoured). Disabling stops camera tracks, closes the
  landmarker, removes the hidden video element, and emits `point-lost`.
- **Open palm → engage:** nothing acts on the gallery until an open palm is seen;
  incidental hand motion while not engaged is inert.
- **Point → curatorial focus:** the index fingertip becomes a smoothed (EMA), mirrored,
  device-neutral `point-at` stream. `gallery/GesturePointTarget` projects wall placements
  through the camera and applies target magnetism (nearest print within a generous
  radius), driving the same focus ring keyboard users see.
- **Pinch → pull from the wall:** a pinch emits the existing `open-focused` intent — the
  focused print nudges off the wall and the detail view opens, semantically identical to
  click/Enter. A refractory period makes one pinch equal one selection.
- **Tracking lost → cancel:** hand absence past a grace period disengages and emits
  `point-lost`; instant intents hold no state, so nothing can be left stuck.
- **Boundaries:** raw landmarks never leave the adapter; the adapter imports nothing
  from gallery/scene/state; MediaPipe loads lazily as its own 41 KB gzip chunk plus
  runtime WASM/model fetched from CDN only when gesture is enabled.

## Verified in this environment (no camera available)

- Unit tests: 8 new (pose classification for open-palm / pointing / pinch / fist with
  synthetic landmark hands; pointer reporting; smoothing behaviour) — 31 total passing.
- **Denied/no-camera path, live:** enabling gesture in a camera-blocked browser showed
  "Camera permission was declined — pointer and keyboard remain fully available.", the
  button returned to its enable state, and the complete keyboard journey (browse →
  detail → back) was re-verified immediately after the denial.
- Main bundle unchanged at 304 KB gzip; gesture chunk loads on demand only.

## Outstanding — requires real hardware and humans (per V2.5-protocol.md)

- Every human exit criterion: discovery without coaching, take/pull language,
  completion/error/accidental-activation measurements, lighting/skin-tone variability.
- Live tracking-loss behaviour with a physical hand; engagement grace-period tuning.
- Frame-rate comparison with gesture on/off on owner hardware.
- **The protocol itself requires owner approval before test runs begin.**

## Risks carried forward

- Classification thresholds (pinch ratio, extension ratio, smoothing α, magnetism
  radius) are set from geometry reasoning and synthetic tests, not from human runs —
  expect tuning after the first protocol session.
- CDN-hosted WASM/model is acceptable for validation; self-hosting is a V5 (publishing)
  concern and is noted for ADR review then.
