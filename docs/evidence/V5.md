# V5 Evidence — Publishing Platform / Lumina Space 1.0

## Delivered journey (verified locally, end to end, 2026-07-16)

`review draft → publish → open stable link anonymously → edit draft → verify live
unchanged → republish → verify link advances → unpublish → 410 → republish → live again`

The same-origin production shape was additionally verified through a local Wrangler Worker:
the built SPA, Hono API, Better Auth session, anonymous public viewer, and `/e/:slug`
navigation all operate on one origin. This is release-candidate evidence, not a claim that
production deployment or cross-device acceptance has happened.

## Repeatable automated evidence

`publicationService.test.ts` exercises the publication service against an in-memory
implementation of the same repository contract used by Drizzle/Neon. It verifies:

- stable slug across first publish, republish, unpublish, and republish restoration;
- draft and library edits cannot mutate the current immutable revision;
- a new revision advances sequence while retaining the old snapshot;
- snapshot rows are written before the stable publication pointer advances;
- workspace ownership and readiness fail before publication state is created;
- unknown and known-removed public states remain distinct.

The earlier real Neon/R2 journey remains provider evidence, including the published-photo
delete guard (API 409 plus database foreign-key backstop). Automated tests do not pretend to
replace that provider-specific storage check.

## ADR 0005 invariant record (e2e run)

| Step | Result |
|---|---|
| Publish | slug `owner-review-exhibition-…`, revision 1 |
| Anonymous public read (no cookies sent) | 200, correct title, 3 photos |
| Edit draft title | public title/revision **unchanged** |
| Draft state | `draftChangedSincePublish: true` |
| Republish | **same slug**, revision 2, edited title now live |
| Delete a published photograph | **409 referenced** (DB-level FK backstop as well) |
| Unpublish | public read **410**; revisions retained |
| Unknown slug | **404** |
| Republish after unpublish | revision 3, public 200 again |

UI verification: editor shows "Live: revision N — up to date / has unpublished
changes", the public link, Publish → Republish → Published ✓ transitions, and
Unpublish. The public page `/e/:slug` renders the revision in the full 3D gallery
(title + description overlay, pointer/keyboard journey verified, gesture opt-in
present, 2D fallback path shared with the main app) with **no account surface at all**.

## Engineering decisions recorded

- **Atomicity without transactions:** neon-http cannot run transactions, so publish
  writes the revision + snapshot rows first and advances the publication pointer as
  the final single UPDATE. A mid-publish failure can only orphan an unreferenced
  revision — the public link never sees a partial state.
- **Database clock everywhere:** `draftChangedSincePublish` compares `updatedAt`
  with `publishedAt`; both now come from Postgres `now()`. The original Worker-clock
  write produced a false negative under clock skew — caught by e2e, fixed the same day.
- **`Cache-Control: no-store`** on the public API: correctness before caching while
  presigned URLs expire hourly and republish must be immediately visible; revisit
  with deployment traffic.
- **Snapshot denormalisation:** revision photo rows copy title/caption/dimensions, so
  a published revision stays exactly what was published regardless of later edits.
- **Routing:** `/e/:slug` is a hand-rolled path match; a router dependency arrives
  with a third route, not before.
- **One Worker, one origin:** `apps/web/dist` is a Cloudflare static-assets binding while
  `/api/*` and `/e/*` run Hono first. The production web build uses an empty API base, so
  browser requests and Better Auth cookies are same-origin; Vite development retains port
  8787 explicitly.
- **Share metadata from the immutable revision:** the Worker reads the stable slug's current
  revision before returning the SPA shell and injects document title, description, Open
  Graph, Twitter, canonical URL, and cover preview. A metadata failure falls back to the
  generic shell rather than making the exhibition unavailable.
- **Visible scene preparation:** the app keeps a warm `Lighting the exhibition…` layer until
  Drei's texture loading manager reports completion and the canvas renders a frame. This
  replaces the observed several-second near-black first impression.
- **Actionable gesture failure states:** camera denial, missing/busy camera, and MediaPipe
  model startup failure now have separate messages; every branch preserves conventional
  viewing.

## Local production-shape verification

- `npm run typecheck`: both workspaces passed.
- `npm test`: API 20 + Web 40 = 60 tests passed.
- `npm run build` builds the Vite app and completes `wrangler deploy --dry-run`.
- Wrangler read 18 static files and bundled the Worker successfully.
- The local Worker served `/`, `/api/*`, and `/e/:slug` from one origin.
- The review account signed in successfully on the Worker origin and loaded its workspace.
- The creator root no longer loads the V1 AI-generated sample gallery behind account controls.
  Signed-in users now land on their named Workspace, where server-owned exhibition drafts
  appear as a cover-led collection using the chosen real cover asset. Missing covers and
  empty collections remain honest states rather than generated placeholders. A cover opens
  its exhibition directly; Draft Preview alone enters the real 3D runtime and Exit Preview
  returns to Workspace. This journey was verified in the local browser.
- The published review exhibition returned revision-specific title, description, URL, and
  signed cover image in the document head before SPA execution.
- Shows with one to six photographs now compose on the visible primary wall; seven or more
  activate multiple walls. Regression tests cover both branches.
- R2 rejected Three.js texture reads from the temporary local Worker origin because the bucket
  CORS policy listed only the Vite origins. The final free production origin is now allowlisted;
  a signed production image request returned `200 image/jpeg` with the exact production
  `Access-Control-Allow-Origin`. A disposable production upload remains an owner-acceptance item.
- Vite still reports the known main-chunk size warning (about 310 KB gzip). No measured
  regression or second route architecture currently justifies speculative bundler work.
